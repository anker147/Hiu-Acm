# 2026-06-27 工作日志

## 移动端登录被拦在 token 校验外（已修复）

### 症状
- PC 端 Chrome 登录正常、保持登录正常
- 移动端浏览器（iOS Safari / 微信 / Android Chrome）登录后立即掉线或后续请求 401

### 根因
- `js/auth.js` 和 `js/api.js` 都用 `sessionStorage` 存 `hiu_token`
- `sessionStorage` 在移动端多个场景会被静默清空：
  1. iOS 私密浏览模式（直接抛错或不可用）
  2. 微信内置浏览器 / X5 内核 切回前台
  3. Android Chrome 切后台被系统回收
  4. 标签页切换 / PWA 启动
- 现象：「登录接口 POST /api/login 成功 → 拿到 token 写入 sessionStorage → 紧接着 `checkSession()` 读不到 token」

### 修复
- `js/api.js`：`setToken`/`init` 改用 `localStorage`
- `js/auth.js`：新增 `tokenStore` 抽象层（get/set/remove），`checkSession`/`logout`/`isAdmin`/`getPhone` 全部走 localStorage
- JWT 后端 24h 过期机制不变，前端持久化与后端过期校验是两层

### 部署
- 这是纯前端变更，**只需 git push**，不需要 wrangler deploy
- 提醒用户：移动端需清浏览器缓存或用无痕模式打开，避免旧 JS 缓存导致看起来没生效

## 新设备无法访问 workers.dev（已用 Pages 反代解决）

### 症状（关键新信息）
- 没登录过系统的设备 → API 完全无响应 + 登录异常
- 登录过的设备 → 一切正常（任何账号都能登）
- 排除了代码、token、CORS 问题

### 根因（Cloudflare workers.dev 国内访问不稳定）
1. **DNS 解析**：新设备首次解析 workers.dev 走运营商 DNS，返回的 Cloudflare 边缘 IP 在国内不可达；老设备 DNS 缓存里有可用 IP
2. **HTTP/3 (QUIC)**：Cloudflare 默认 alt-svc h3:443，新设备首次走 UDP 443 被运营商丢包，5-10s 才 fallback 到 HTTP/2；老设备有 HTTP/2 长连接复用
3. **TLS ECH**：Cloudflare 启用 Encrypted Client Hello，新设备首次 TLS 握手可能被丢包

### 解决方案：Cloudflare Pages 反代（已实施）

**架构变更**：
```
新设备 → https://hiu-acm.pages.dev/ (前端同源)
         ↓ /api/* 请求
         Pages Function (functions/api/[[path]].js)
         ↓ fetch 转发
         原 Worker (https://hiu-acm-api.wangyiyouxiang0719.workers.dev)
         ↓
         D1 数据库
```

**改动**：
1. 新增 `functions/api/[[path]].js` — Pages Function catch-all 反代
2. 修改 `js/api.js` API_BASE：
   - 生产环境（pages.dev）→ `/api`（同源相对路径，走 Pages Function）
   - 本地调试（localhost/127.0.0.1）→ 直连 Worker

**部署**：
```bash
npx wrangler pages deploy . --project-name=hiu-acm --branch=main
```

**验证通过**：
- `GET /` → 200
- `GET /api/health` → `{"status":"ok","db":true,...}` ✅ 反代正常
- `POST /api/login` → 业务错误「用户不存在」 ✅ 反代 + Worker 链路完整
- `OPTIONS /api/login` → 204 + 完整 CORS 头 ✅

**新访问地址**：`https://hiu-acm.pages.dev`

### 待办

- 如果还有问题，可能 pages.dev 在该设备/网络下也不可达，需考虑备案自定义域名

### 部署机制提示

## Bug：API_BASE 双层 `/api` 前缀（已修复）

### 症状
- 部署 Pages 反代后，登录报 401，浏览器 Console 显示 `/api/api/login` 401
- 路径里多了一层 `/api`

### 根因
- 前端调用代码 `this.fetch("/api/login")` 写的是完整路径
- `API_BASE` 设为 `"/api"` 后拼接成 `"/api" + "/api/login"` = `/api/api/login`
- Pages Function 把 `/api/api/login` 转发到 Worker → Worker 没这个路由 → 落到 auth 校验 → 401

### 修复
- `API_BASE` 改为空字符串 `""`（生产环境）
- 前端调用 `/api/login` → `API_BASE + "/api/login"` = `"" + "/api/login"` = `/api/login`（正确）
- 本地调试仍走 Worker 完整 URL
- `index.html` 加版本号 query string `?v=20260627b` 防缓存
- 加 meta http-equiv Cache-Control no-cache

### 教训
- Pages Functions 的 catch-all 路由是 `/api/<path>`，前端**要么**用 `API_BASE=""` + 调用 `/api/xxx`，**要么**用 `API_BASE="/api"` + 调用 `/xxx`。不能两边都带 `/api`
- 选前者更直观，因为前端调用代码已经是 `/api/xxx` 形式

### 验证通过
- `POST /api/login` → `{"error":"用户不存在"}` (404 业务错误，说明路由命中)
- `GET /api/health` → `{"status":"ok"}` ✅
- `POST /api/api/login` → `{"error":"未登录或令牌已过期"}` (401，旧错误路径已不命中)

## Bug：用户登录日志不写入（已修复）

### 症状
- 用户登录后管理端「登录日志」页面看不到任何记录
- 仅 admin 自己历史的少数记录可见，device/region 为空

### 根因（migration 002 没跑过）
- `workers/api.js` 第 882 行登录成功后执行：
  `INSERT INTO login_logs (phone, ip, user_agent, login_at, region, device) VALUES (?, ?, ?, ?, ?, ?)`
- 但生产 D1 数据库的 `login_logs` 表只有 5 列（id/phone/ip/user_agent/login_at），**没有 `region` 和 `device` 列**
- migration `002_login_logs_region.sql` 从来没执行过
- INSERT 报错 `no such column: device` → 被 `try/catch` 静默吞掉（注释是「日志记录失败不影响登录返回」）
- 表面：登录成功；实际：日志完全没写入

### 修复
- 跑 `npx wrangler d1 execute hiu-acm-db --remote --file=migrations/002_login_logs_region.sql`
- 验证：表结构已有 7 列（含 region/device）
- 用 admin 账号登录测试 → id 118 写入成功，device="Windows电脑"，region="香港-油尖旺區-旺角" ✅

### 教训（重要）
- **改后端代码引入新列后，必须立即跑 migration**！否则生产数据库缺列，所有 INSERT 静默失败
- 这个 bug 在 MEMORY.md 的「部署注意」里早就写过（"改后端表结构后必须先跑 migration 再部署 Worker"），但实际执行时漏了
- 后续如果再改 Worker 涉及表结构，**必须三步走**：
  1. 写 migration SQL 文件
  2. 跑 `npx wrangler d1 execute hiu-acm-db --remote --file=migrations/xxx.sql`
  3. 部署 Worker `npx wrangler deploy workers/api.js --env=""`
  4. 缺一不可，且顺序必须是 migration → deploy Worker

## 登录日志增强：浏览器指纹 + IP 透传修复（已实施）

### 用户痛点
1. 看不到其他用户的登录日志（实际上是因为 migration 没跑导致 INSERT 失败，已修）
2. 日志解析有问题：device 只显示「Windows电脑」等粗略信息，无浏览器/OS 版本
3. 没有浏览器指纹，无法识别同一设备的不同账号 / 同一账号的不同设备
4. 所有日志 IP 都是 Cloudflare Pages 节点 IP（`2a06:98c0:3600::103`），不是真实用户 IP

### 改动

**1. 新增浏览器指纹采集 `js/fingerprint.js`**
- 采集维度：UA / 平台 / 屏幕 / 时区 / 语言 / Canvas 哈希 / WebGL 厂商
- 输出：16 位 SHA-256 哈希 `fingerprint` + 完整字段 `fpDetail`
- 在 `Auth.login` 内部自动采集，调用方无感

**2. 前端 `js/auth.js` + `js/api.js`**
- `Auth.login(phone, code)` 内部先 `Fingerprint.collect()`，再调 `Api.login(phone, code, { fingerprint, fpDetail })`
- `Api.login` 把指纹放进请求 body

**3. 后端 `workers/api.js`**
- 新增 `parseUADetail(ua)` 返回 `{ os, browser, device, summary }`，比 `parseUA` 详细（含版本号）
- `ipToLocation` 加 fallback 链：ip-api.com（中文）→ ipapi.co（英文）→ "未知"
- 登录日志 INSERT 增加 `fingerprint`/`fp_detail`/`ua_parsed` 三列
- 异常不再静默吞，改 `console.error("[login_logs] INSERT 失败:", ...)` 便于排查
- `enrichLog` 返回新字段，历史 `ua_parsed` 空值按需回填

**4. Migration 003**
- `login_logs` 加 `fingerprint TEXT`、`fp_detail TEXT`、`ua_parsed TEXT` 三列
- 已跑 `npx wrangler d1 execute hiu-acm-db --remote --file=migrations/003_login_logs_fingerprint.sql`

**5. Pages Function 反代 IP 透传修复（重要）**
- 问题：Cloudflare 在 Pages→Worker 转发时**会重写 `CF-Connecting-IP`**，导致上游 Worker 拿到的是 Pages 节点 IP（如 `2a06:98c0:3600::103`）而非真实用户 IP
- 解决：`functions/api/[[path]].js` 在转发时把真实 IP 放进自定义头 **`X-Real-Client-IP`**（Cloudflare 不会重写自定义头）
- Worker 端取 IP 优先级：`X-Real-Client-IP` > `CF-Connecting-IP` > `X-Real-IP` > `X-Forwarded-For`
- 验证：用 `/api/debug-headers` 临时端点确认 `x-real-client-ip: 103.151.172.96`（curl 真实 IP），`cf-connecting-ip: 2a06:98c0:3600::103`（节点 IP）

**6. 管理端 `js/admin.js`**
- `renderLogs` 表格加「指纹」列、「设备/浏览器」列（用 `uaParsed` 代替粗略的 `device`）
- 详情展开显示完整指纹 + 指纹详情 JSON（折叠面板）
- `viewUserLogs` 同步加指纹列
